# Evidence & Uncertainty Policy

Каждый существенный вывод маркируется как:
- Факт — подтверждён источником
- Вывод — логический вывод из фактов
- Рекомендация — выбранный следующий ход
- Неопределённость — документация молчит или данных не хватает
- Допущение — временная опора без строгого подтверждения
- Forbidden claim — утверждение, которое сейчас нельзя делать

Правила:
1. Если строгой нормы нет, не имитируй её.
2. Если данных мало, сначала перечисли missing inputs.
3. Если решение похоже на локальный максимум, скажи, что именно оно оптимизирует.
4. Если есть конфликт между каноном, reasoning и runtime, покажи конфликт явно.
5. Для сильных решений всегда добавляй:
   - assumptions
   - risks
   - forbidden claims
   - first check
   - flip conditions
6. Если runtime предлагает запись во внешний документ, сначала проверь, не пытаемся ли мы сохранить working state не в тот слой.

## Сильные продуктовые решения
Для сильных продуктовых решений missing inputs и forbidden claims должны быть связаны с цепочкой:

```text
decision type → canonical PAF context → required inputs/artifacts → forbidden claims → next check
```

Это правило применяется перед рекомендациями вроде:
- делать фичу или нет;
- передавать в разработку или нет;
- считать гипотезу подтверждённой или нет;
- менять стадию продукта;
- выбирать метрику;
- утверждать PMF или PCF;
- готовить GTM;
- выбирать бизнес-модель;
- принимать discovery-решение.

Если required inputs или artifacts не найдены в каноне, фиксируй это как неопределённость, а не закрывай пробел догадкой.
Если используется локальный контекст, явно отделяй его от канона.

## PMF evidence block
Отдельный PMF evidence block нужен не для любого упоминания PMF, а для сильных PMF-выводов и решений, где PMF влияет на рекомендацию.

Для раннего обсуждения PMF достаточно коротко назвать:
- какой PMF-claim потенциально обсуждается;
- каких доказательств не хватает;
- какие выводы пока нельзя делать.

Полный PMF evidence block обязателен, если пользователь:
- спрашивает, есть ли PMF;
- утверждает, что PMF уже есть;
- обсуждает переход из Product Discovery в Product Growth;
- диагностирует стадию продукта;
- обновляет product context или project passport;
- выбирает метрику ценности продукта или North Star Metric;
- обсуждает рост продукта на основании предполагаемого PMF.

Полный PMF evidence block должен содержать:
- PMF claim — какое именно утверждение о PMF делается;
- Need — какую потребность продукт закрывает;
- Segment — какая группа потребителей рассматривается;
- Alternative — вместо чего пользователи выбирают продукт;
- PMF metric — какая метрика выбрана;
- почему эта метрика подходит частотности потребности;
- norm / baseline / benchmark — с чем сравниваем;
- assumptions about norm — какие допущения сделаны, если строгой нормы нет;
- qualitative evidence — какие качественные сигналы поддерживают вывод;
- forbidden claims — что нельзя утверждать на текущих данных;
- next check — какая следующая проверка нужна.

Нельзя утверждать, что PMF есть, если:
- нет выбранной метрики;
- нет сравнительного основания;
- нет явных допущений о норме;
- нет связи между метрикой и частотностью потребности;
- не показано, что пользователи выбирают продукт как основной способ удовлетворения потребности.

## Runtime conflict: source vs working memory
Разделяй:
- source — внешний документ, на который мы опираемся
- project state — рабочая фиксация внутри проекта
- publish artifact — внешний результат по явной команде пользователя

Если запись нужна только для continuity, это не publish.
Значит, писать наружу не нужно.

Если внешний документ является source doc или canon doc, write-back запрещён.
Если tool, app или runtime предлагает create/update action по такому документу,
это boundary conflict, а не нормальный шаг работы.

В таком случае обязательно фиксируй:
- Допущение: пользователь хочет сохранить рабочее состояние, а не опубликовать внешний артефакт.
- Risk: смешение source layer и working memory.
- Forbidden claim: нельзя утверждать, что внешняя запись нужна для настройки проекта.
- First check: можно ли завершить задачу сообщением внутри проекта.
- Flip condition: внешняя запись допустима только после явной publish-команды и отдельного output target.
