# How to Set Up a CPO Copilot Project

Назначение
Эта папка помогает быстро запустить CPO copilot в новом GPT Project.

Важно
Этим copilot может пользоваться не только CPO.
Он подходит и инженеру, и аналитику, и дизайнеру, и маркетологу, если им нужен сильный продуктовый собеседник.

Важно 2
Copilot умеет работать в двух ситуациях:
1. Когда продукт или проект уже есть
2. Когда продукта пока нет, и человек только выбирает, что вообще стоит делать

Чтобы не ломать методологию, эти две ситуации разведены:
- если продукт уже есть, собирается product context;
- если продукта ещё нет, собирается exploration context.

## Что должно быть подключено
В рабочей папке должны лежать:
1. [CANON] PAF Knowledge Layer.pdf
2. [METHOD] CPO Copilot UX Kernel
3. [METHOD] Evidence & Uncertainty Policy
4. [METHOD] PAF Answer Modes
5. CPO Copilot Project Setup

## Что лежит в setup-папке
1. [START HERE] Activate CPO Copilot
2. [METHOD] CPO Copilot Project Instruction Template
3. [METHOD] Recommended Product Context Schema for CPO Copilot
4. [METHOD] Recommended Exploration Context Schema for CPO Copilot
5. [TEMPLATE] Product Context Note Template for CPO Copilot
6. [TEMPLATE] Exploration Context Note Template for CPO Copilot
7. [CHECKLIST] CPO Copilot Setup Checklist

## Что добавлять в Sources обязательно
### Во всех случаях
- рабочую runtime-папку;
- setup-папку CPO Copilot Project Setup, если она не входит в runtime-папку.

### Если продукт уже есть
Добавляй:
- [PROJECT NOTE] Текущий продуктовый контекст, если уже существует;
- актуальные продуктовые документы;
- актуальные исследования;
- актуальные метрики, dashboards или analytics;
- decision log, если он есть.

### Если продукта пока нет
Добавляй:
- [PROJECT NOTE] Текущий контекст поиска направления, если уже существует;
- заметки об идеях, проблемах, сигналах, интервью, наблюдениях;
- материалы о рынке, если они уже есть;
- заметки об ограничениях, доступах, сильных сторонах и возможных аудиториях.

## Что не добавлять в Sources по умолчанию
Не добавляй:
- всю внутреннюю мастерскую целиком, если нужна только рабочая настройка;
- live-буферы и сырые внутренние заметки по развитию метода;
- candidate-слой без явной причины;
- raw-export архивы и шумные выгрузки без связи с текущей задачей;
- устаревшие документы без пометки, что они устарели;
- большие папки «на всякий случай».

## Как запускать

### Режим A — базовый
Шаги:
1. Создай новый GPT Project.
2. Добавь как source ссылку на рабочую папку в Google Drive.
3. В первый чат вставь activation prompt.
4. Пройди короткий онбординг.
5. Сохрани итоговую заметку:
   - либо [PROJECT NOTE] Текущий продуктовый контекст
   - либо [PROJECT NOTE] Текущий контекст поиска направления

Activation prompt

Настрой продуктового копилота. Сначала найди в project sources файл [START HERE] Activate CPO Copilot и следуй ему буквально. Если не найден один из основных рабочих файлов, скажи прямо, чего не хватает. Сначала определи, работаем ли мы с уже существующим продуктом или пока только выбираем направление. Потом прямо скажи:
- что уже подключено в Sources;
- что обязательно добавить в Sources сейчас;
- что не стоит добавлять;
- что можно добавить позже;
- чего пока не хватает.
Не переходи к проектированию раньше времени. В конце собери нужную рабочую заметку.

Очень короткий prompt

Используй этот вариант, если в Project settings уже вставлен файл
[METHOD] CPO Copilot Project Instruction Template

Тогда в первый чат достаточно написать:

Настрой продуктового копилота



### Режим B — усиленный
Шаги:
1. Создай новый GPT Project.
2. Добавь как source ссылку на рабочую папку в Google Drive.
3. Открой Project settings.
4. Вставь туда текст из файла [METHOD] CPO Copilot Project Instruction Template
5. В первый чат отправь:
   Настрой CPO копилота

## Что должно произойти после активации
1. Copilot должен назвать, какие источники уже найдены.
2. Copilot должен определить:
   - product mode
   - exploration mode
3. Copilot должен прямо сказать:
   - что обязательно добавить в Sources;
   - что не стоит добавлять;
   - что можно добавить позже.
4. Copilot должен отделить общий методический слой от локального контекста.
5. Copilot должен показать, чего не хватает.
6. Copilot должен задать один лучший следующий вопрос.
7. Copilot должен собрать черновик нужного типа заметки.
8. Copilot должен прямо сказать, когда дальше уже начинается серьёзное проектирование.
9. Copilot не должен переходить туда без явного сигнала пользователя.

## Что делать после онбординга
### Если был product mode
Сохрани:
[PROJECT NOTE] Текущий продуктовый контекст

### Если был exploration mode
Сохрани:
[PROJECT NOTE] Текущий контекст поиска направления

Потом:
- либо продолжай работать в том же режиме;
- либо переходи из exploration mode в product mode, когда объект работы уже достаточно ясен.
